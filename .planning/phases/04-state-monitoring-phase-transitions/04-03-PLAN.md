  <action>
    1. Add `stateWatchDebounceMs` config field to AutopilotConfigSchema in config.ts:
       - Type: z.number().int().min(100).default(500)
       - Purpose: configurable debounce for STATE.md watcher

    2. In daemon.ts `runDaemon()`, for each goal in the processing loop:
       - Before calling orchestrateGoal(), construct the STATE.md path: `path.join(config.workspaceRoot, '.planning', 'STATE.md')`
       - Create a StateWatcher instance with the path, debounce from config, and a child logger
       - Register event listeners on the watcher:
         - 'state_changed': log at debug level with snapshot summary
         - 'phase_advanced': log at info level with from/to phase numbers
         - 'plan_advanced': log at info level with phase and plan numbers
         - 'phase_completed': log at info level with phase number and name
         - 'goal_completed': log at info level
       - Call watcher.start() before orchestrateGoal()
       - Call watcher.stop() in a finally block after orchestrateGoal() completes (success or failure)

    3. On shutdown: if watcher is running, stop it during the graceful shutdown path. The finally block in the goal loop handles this naturally.

    Do NOT make state watching mandatory — if StateWatcher construction fails (e.g., parent dir doesn't exist), log a warning and proceed without watching. The orchestrator must still work without a watcher.
  </action>
  <verify>npx tsc --noEmit passes. Daemon compiles with StateWatcher integration.</verify>

    3. In daemon.ts, when calling orchestrateGoal(), pass an onProgress callback that:
       - Logs the snapshot at debug level
       - This provides a polling-based complement to the event-based StateWatcher

    Keep the existing orchestrator flow unchanged — onProgress is purely additive observability, not control flow.
  </action>
