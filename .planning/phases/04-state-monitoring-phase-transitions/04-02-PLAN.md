  <action>
    Install chokidar: `npm install chokidar`

    Create `src/state-watcher.ts` with a `StateWatcher` class that:

    1. Constructor accepts: `{ stateMdPath: string, debounceMs?: number, logger: Logger }`
       - Default debounceMs: 500
    2. Extends Node.js EventEmitter (import from 'node:events')
    3. `start()` method:
       - Creates chokidar watcher on stateMdPath with `{ persistent: true, ignoreInitial: false }`
       - On 'add' and 'change' events: debounce, then read file and parse with readStateMd()
       - Store latest StateSnapshot as `this.lastSnapshot`
       - On first successful parse: emit 'ready' event with snapshot
    4. `stop()` method:
       - Closes chokidar watcher
       - Removes all listeners
    5. `getLastSnapshot()` method: returns current StateSnapshot or null
    6. Debounce implementation: use a simple setTimeout/clearTimeout pattern — do NOT add lodash or another dependency for this. Clear pending timeout on each file change, set new one with debounceMs delay.
    7. Handle file not existing yet: chokidar watches parent directory until file appears. Log at debug level when file is missing, info when first detected.
    8. Handle read errors gracefully: log warning, keep last known snapshot.

    Export: StateWatcher class, StateWatcherOptions interface.
  </action>
  <verify>npx tsc --noEmit passes with no errors</verify>
  <done>StateWatcher class compiles, has start/stop lifecycle, debounces file changes, parses STATE.md on each change</done>

<task type="auto">
  <name>Task 2: Add transition detection and typed progress events</name>
  <files>src/state-watcher.ts</files>
  <action>
    Extend StateWatcher with transition detection by comparing previous and current snapshots:

    1. Define ProgressEvent types (use a discriminated union or string literal type):
       - `state_changed`: { previous: StateSnapshot | null, current: StateSnapshot } — emitted on every parsed change
       - `phase_advanced`: { fromPhase: number, toPhase: number, phaseName: string } — phaseNumber increased
       - `plan_advanced`: { phaseNumber: number, fromPlan: number, toPlan: number } — planNumber increased within same phase
       - `phase_completed`: { phaseNumber: number, phaseName: string } — status contains "complete" (case-insensitive)
       - `goal_completed`: { progressPercent: number } — status is "Complete" and progressPercent is 100 (or totalPhases === phaseNumber and status contains "complete")

    2. In the debounced change handler, after parsing:
       - Compare new snapshot with this.lastSnapshot
       - Detect and emit applicable transition events BEFORE updating this.lastSnapshot
       - Update this.lastSnapshot after event emission
       - Log each emitted event at info level with structured data

    3. Comparison logic:
       - phase_advanced: current.phaseNumber > previous.phaseNumber
       - plan_advanced: current.phaseNumber === previous.phaseNumber && current.planNumber > previous.planNumber
       - phase_completed: current.status matches /complete/i && (previous was null OR previous.status did NOT match /complete/i OR previous.phaseNumber !== current.phaseNumber)
       - goal_completed: current.phaseNumber === current.totalPhases && current.status matches /complete/i (only emit once)

    4. Add a `goalCompleteEmitted` flag to prevent duplicate goal_completed events.

    Export: ProgressEvent type, individual event type interfaces.
  </action>
  <verify>npx tsc --noEmit passes with no errors</verify>
  <done>StateWatcher emits typed progress events on state_changed, phase_advanced, plan_advanced, phase_completed, and goal_completed transitions. Transition detection compares snapshots correctly.</done>
