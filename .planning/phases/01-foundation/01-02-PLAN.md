<task type="auto">
  <name>Task 2: Parse goals markdown into an execution plan</name>
  <files>src/goals.ts, goals.md</files>
  <action>
    Implement resilient goals parsing and execution-plan building:
    - Treat `goals.md` as a structured queue with three sections: `## pending`, `## in progress`, and `## done`.
    - For each section, parse markdown checklist items of the form `- [ ] Title` or `- [x] Title` and map them to a `Goal` object with `title`, `status`, and `raw` line.
    - Normalize newlines and ignore any content before the first recognized section header.

    Add lightweight annotations extracted from square-bracket tokens embedded in the line:
    - `[group:alpha]` → `parallelGroup: "alpha"`.
    - `[after:beta,gamma]` → `dependsOn: ["beta","gamma"]`.
    - `[priority:1]` → numeric `priority: 1`.
    Strip these tokens from the human-facing `title` but preserve them in the structured fields.

    Implement helpers:
    - `parseGoals(markdown: string): Goal[]` that handles normalization and annotation parsing.
    - `loadGoals(filePath: string)` that reads the file via `fs/promises`, throws a clear error if missing, tolerates empty files by returning an empty array, and warns when there are no pending goals.
    - `getPendingGoals(goals)` to filter by `status === 'pending'`.
    - `buildExecutionPlan(goals)` that sorts goals into an `ExecutionPlan` by:
      - Primary key: explicit numeric `priority` (ascending; missing priority maps to `Number.MAX_SAFE_INTEGER`).
      - Secondary key: original order in the file (stable sort).
      Attach `parallelGroup` and `dependsOn` metadata to plan items without enforcing any scheduling semantics yet.
  </action>
  <verify>
    - Add or update tests in `src/goals.test.ts` (or equivalent) to cover:
      - Section-based status assignment and basic checkbox parsing.
      - Annotation parsing for `group`, `after`, and `priority` tokens.
      - Execution-plan ordering for mixed-priority goals.
    - Run `npm test` and confirm all goals-related tests pass.
    - Run `npm run build` followed by `node dist/cli.js --dry-run --goals ./goals.md` and confirm it prints a queue that matches the parsed `goals.md`.
  </verify>
  <done>
    - Goals markdown in `goals.md` round-trips through `loadGoals` and `buildExecutionPlan` with correct statuses and ordering.
    - Missing or empty `goals.md` produces a clear error or an empty queue without crashing the daemon.
    - Dry-run CLI output reflects the parsed goals accurately, including pending count.
  </done>
</task>
