<objective>
Expose the GSD daemon's progress and health via an HTTP status server with graceful shutdown. CLI and status endpoint only for v1 — no full web UI required beyond a minimal dashboard.

Purpose: Provide a way to see what the autopilot is doing (goal, phase/plan, resume status, recent activity) without reading logs or STATE.md by hand.
Output: Express-based status server wired into the daemon lifecycle: GET /status (legacy JSON), GET /api/status (rich payload), GET /api/config and POST /api/config for parallelization toggle, graceful shutdown on daemon exit.
</objective>

Ensure the status server implementation is complete and correct:
- GET /status returns legacy JSON: { running, currentGoal?, phaseNumber?, planNumber?, heartbeat? }
- GET /api/status returns rich payload: stateSnapshot, sessionLogEntries, gitFeed, systemLoad when options (stateMdPath, sessionLogPath, workspaceRoot) are provided
- createStatusServer handles EADDRINUSE non-fatally: returns { server: null, close: no-op }, does not throw
- Daemon starts status server when config.statusServerPort is set; wraps createStatusServer in try/catch so startup failures do not crash the daemon
- On shutdown, daemon awaits statusServerClose() so the HTTP server closes cleanly (no open handles)
- When options.planningConfigPath is set, GET /api/config returns merged planning config; POST /api/config validates and persists parallelization slice to .planning/config.json
Run: npm test -- status-server
