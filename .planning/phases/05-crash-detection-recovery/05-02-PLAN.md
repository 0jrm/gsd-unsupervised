  <name>Task 1: Compute resume pointer from session log and STATE.md</name>
  <files>src/resume-pointer.ts, src/session-log.ts, src/state-parser.ts, src/types.ts</files>
  <action>
Add a new `src/resume-pointer.ts` module responsible for determining where to resume from on startup.

Design:
- Export a pure function `computeResumePointer(opts)` that:
  - Loads the session log via `loadSessionLog()` and, if available, the latest STATE snapshot via `readStateMd`.
  - Treats missing or unreadable session log/STATE as "no resume" (return `null`).
  - Derives the last known successful execution point from the log: the latest `plan-complete` event, or, if none exist, the latest `phase-complete`, falling back to null.
  - Optionally cross-checks STATE.md: if STATE disagrees with the log in a way that indicates an incomplete or inconsistent run (e.g. STATE shows a phase in progress but the log has no corresponding completion event), favor the more conservative pointer (earlier phase/plan) to avoid skipping work.
- Represent the result as a typed `ResumePointer` (e.g. `{ phaseNumber, planNumber } | null`), where `planNumber: 0` means "start at the first plan of this phase".

Keep this module free of side effects: it should not mutate files or start processes; it purely inspects prior state and returns a pointer for the caller to act on.
  </action>
  <verify>npm test</verify>
  <done>`computeResumePointer` exists, returns sensible pointers or null based on synthetic logs/STATE, and is covered by focused unit tests</done>

Changes:
- In `src/daemon.ts` (or equivalent process entrypoint), before starting a new orchestrator run for a goal, call `computeResumePointer` with the workspace/config context and inspect the result.
- If `ResumePointer` is `null`, start from the beginning of the roadmap as today.
- If `ResumePointer` is non-null, start the phase/plan loop from the indicated `phaseNumber` and `planNumber`:
  - Skip phases and plans that have clearly completed in the session log.
  - Ensure any user-facing logging makes it explicit that a resume is happening (e.g. "Resuming from phase 3 plan 2 due to previous crash").
- Keep behavior safe in YOLO mode: do not re-run plans that already have a successful `plan-complete` entry unless the user explicitly opts out in future phases (this plan only implements the default conservative behavior).

Ensure the integration:
- Does not break existing orchestrator tests (add or adjust tests to cover the new starting offset behavior).
- Keeps crash detection/resume logic in a narrow surface area (daemon/orchestrator entry) so that future phases can add CLI flags or configuration without rewriting the core loop.
  </action>
