**Tech stack available:** TypeScript, Zod, Node streams and readline, vitest.
**Established patterns:** Discriminated-union Zod schemas for Cursor NDJSON events, helper functions for extracting session IDs and result events from an in-memory event array.
**Constraining decisions:**
- The Cursor CLI emits one JSON object per line with a `type` discriminator (`system`, `assistant`, `tool_call`, `result`), and plans are executed in non-interactive, auto-approved mode.
- Stream parsing code should tolerate malformed lines and unknown event shapes without throwing.
- Completion detection relies on a final `result` event plus the process exit code and timeout flag from `runAgent`.

**Issues being addressed:** Reduce brittleness around NDJSON parsing and make it easy for the orchestrator to interpret Cursor output when determining plan success/failure.
