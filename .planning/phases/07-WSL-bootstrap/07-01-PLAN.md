**Phase goal from roadmap:**
Phase 7 focuses on WSL bootstrap: setup script and WSL-specific path resolution for Cursor, `clip.exe`, and workspace paths, on top of the already-completed status server and crash recovery work.

**Assumptions and constraints:**
- Tech stack is Node ≥18 with TypeScript and existing CLI/config/logging stack (Commander, chokidar, simple-git, pino, express, zod).
- The CLI already knows the workspace root and has a configuration layer for paths and agent invocation.
- Parallelization is disabled in `.planning/config.json`, so this plan can assume sequential execution with no need for plan-level dependency optimization.

**High-level design for this phase:**
- Add a small WSL/OS detection helper that can answer: "Am I in WSL?", "What is the Windows mount root?", and "What is the corresponding Windows path for this WSL path?"
- Add a centralized path resolution module that can derive:
  - The Cursor CLI/agent path used by the project (defaulting sensibly and allowing overrides from config/env).
  - The Windows `clip.exe` path when in WSL (e.g., `C:\\Windows\\System32\\clip.exe`) and a no-op fallback when unavailable.
  - Normalized workspace paths suitable for logging and for passing into external tools (Cursor agent, clipboard bridge).
- Add a bootstrap entry for WSL that wires these helpers into the existing configuration/state so later phases/plans can call one place for environment-aware paths.
</context>

  <name>Task 1: Add WSL and platform detection helpers</name>
  <files>src/config/wsl.ts</files>
  <action>
Create a small `src/config/wsl.ts` module that exposes:
- `isWsl(): boolean` — detects WSL 1/2 by combining `process.env.WSL_DISTRO_NAME`, `/proc/version` contents, and `process.platform === "linux"`.
- `isWindows(): boolean` — true when `process.platform === "win32"`.
- `toWindowsPath(wslPath: string): string | null` — best-effort conversion of a WSL path like `/mnt/c/Users/...` to a Windows-style path like `C:\\Users\\...`:
  - Handle only `/mnt/<drive-letter>/...` patterns; for non-matching paths, return `null` and let callers decide.
  - Normalize slashes and drive letters correctly (uppercase drive letter + colon).
- `getWindowsRoot(): string | null` — returns `/mnt/c` when on typical WSL installs, or `null` otherwise.

Implementation details:
- Avoid adding new dependencies; rely on the Node stdlib and `fs`/`os` if needed.
- Keep detection conservative to avoid mis-detecting vanilla Linux as WSL: check for at least one of `WSL_DISTRO_NAME`, `/proc/version` containing "Microsoft", or similar.
- Export pure functions with no side effects so they are easy to test and safe to import from anywhere.
  </action>

  <name>Task 2: Implement centralized path resolution for Cursor, clip.exe, and workspace</name>
  <files>src/config/paths.ts</files>
  <action>
Create or extend a `src/config/paths.ts` module that consumes the WSL helpers and centralizes discovery of:
- `getCursorBinaryPath(config: ProjectConfig): string`:
  - Prefer an explicit path from config or env (e.g. `GSD_CURSOR_BIN` / config.json setting).
  - Fallback to a sensible default such as `cursor` on PATH.
  - When running under WSL and the configured path is a WSL path under `/mnt/*`, use `toWindowsPath` to derive a Windows version if the agent must be executed via a Windows binary; otherwise keep the POSIX path for native WSL binaries.
- `getClipExePath(): string | null`:
  - When `isWsl()` is true, return `C:\\Windows\\System32\\clip.exe` (or equivalent) if `toWindowsPath` suggests `C:` is mounted; if not, return `null`.
  - When not on WSL, return `null` so the rest of the app can gracefully skip clipboard integration.
- `getWorkspaceDisplayPath(workspaceRoot: string): { wslPath: string; windowsPath: string | null }`:
  - Always include the original WSL path as `wslPath`.
  - When `isWsl()` is true and the path is under `/mnt/*`, include a `windowsPath` using `toWindowsPath`; otherwise, set it to `null`.

Implementation details:
- Import the new helpers from `src/config/wsl.ts`.
- Keep the API surface small and focused: avoid leaking Node-specific details to the rest of the app; return plain strings and simple objects.
- Do not hardcode workspace paths; assume the caller passes the current workspace root from existing config/state.
  </action>
