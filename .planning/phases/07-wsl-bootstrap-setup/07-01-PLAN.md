**Phase focus:** Phase 7 — WSL Bootstrap & Setup for GSD Autopilot on WSL2.

**Assumptions:**
- Project is running on WSL2 with a mounted Windows C: drive at `/mnt/c`.
- The user's Windows `.cursor` directory is at `/mnt/c/Users/$WIN_USER/.cursor`.
- GSD rules already live in the Windows `.cursor/rules/` directory.

**Constraints:**
- Script must be safe to re-run (idempotent) and avoid destructive operations.
- Script must not assume `sudo` is available or desired for everyday bootstrap steps.
- Script should degrade gracefully with clear error messages when assumptions fail.
</context>

  Create or update a POSIX-compatible `setup.sh` at the repo root that:
  - Uses `#!/usr/bin/env bash` and `set -euo pipefail` with a simple `log()` helper for user-facing messages.
  - Detects WSL2 by checking `/proc/sys/kernel/osrelease` for "microsoft-standard-WSL2" (case-insensitive) and exits with a clear error message if not running under WSL.
  - Detects whether `/mnt/c` is mounted and readable; if not, emits a clear message that Windows integration is required and exits non-zero.
  - Derives a candidate Windows username by inspecting `$WIN_HOME` (if set) or `$USERPROFILE` (if available via `/mnt/c/Users` path), but does not hard-code any specific user name.
  - Prints a short summary of detected environment (WSL version, presence of `/mnt/c`, candidate Windows user path) for debugging.
  Avoid: baking in this specific machine's username or any hard-coded absolute Windows path that will break for other users.
  </action>
  <verify>
  Run `bash -n setup.sh` to ensure the script is syntactically valid, then run `bash setup.sh --check-env` (or similar dry-run flag) to print detection results without making changes, verifying that it detects WSL and `/mnt/c` correctly.
  </verify>
  <done>
  `setup.sh` exists, is executable or runnable via `bash setup.sh`, exits non-zero with a clear message when not on WSL2, and prints a concise environment summary when invoked in check/dry-run mode.

  </verify>
  <done>
